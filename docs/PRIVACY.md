# Приватность и обезличивание медицинских данных

Медицинские изображения и метаданные могут содержать персональные данные. Публичный GitHub‑репозиторий, issue, pull request, лог или скриншот следует считать общедоступным пространством.

## Что уже сделано в этом репозитории

- реальные DICOM‑файлы не включены;
- демонстрационные изображения генерируются алгоритмически и не относятся к человеку;
- `.gitignore` блокирует распространённые расширения медицинских изображений и каталоги исследований;
- `scripts/check_release_privacy.py` проверяет запрещённые расширения и сигнатуру `DICM` перед публикацией;
- логи приложения и локальные экспорты исключены из Git.

Эти меры снижают риск случайной публикации, но не заменяют профессиональное обезличивание.

## Почему удаления ФИО недостаточно

Идентификаторы могут находиться:

1. в стандартных DICOM‑тегах: Patient Name, Patient ID, Birth Date, Accession Number и других;
2. в приватных тегах производителя;
3. в UID, датах исследования и названиях файлов/папок;
4. в burned‑in annotations — тексте, который является частью пикселей;
5. в логах, скриншотах, экспортированных PNG/JPEG и резервных копиях;
6. в истории Git даже после удаления файла из последнего коммита.

## Безопасный процесс публикации примера

Если пример DICOM действительно необходим:

1. Используйте специально созданный синтетический набор данных или открытый учебный датасет с явной лицензией.
2. Обрабатывайте копию, никогда не оригинал исследования.
3. Удалите прямые и косвенные идентификаторы, приватные теги и ненужные UID.
4. Визуально проверьте каждый кадр на текст внутри пикселей.
5. Проверьте имена файлов, архивов, каталогов и метаданные экспортированных изображений.
6. Получите независимую проверку специалиста по ИБ/защите данных вашей организации.
7. Только после этого добавляйте файл в отдельный релиз или хранилище данных с подходящей лицензией.

Не используйте `.gitignore` как доказательство обезличивания: файл без расширения или уже добавленный в Git может обойти это правило.

## Проверка перед релизом

```powershell
python scripts/check_release_privacy.py
git status --short
git ls-files
```

Отдельно просмотрите `assets/screenshots/`, историю коммитов и содержимое будущего архива релиза. Не публикуйте папку `logs/`: пути к локальным исследованиям также могут быть чувствительной информацией.
